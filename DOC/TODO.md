# GPON Diagnostic Framework — TODO & Optimization Ideas

> Generated by DevOps deep-dive analysis. No changes made to codebase.

---

## High Priority (Production Impact)

### 4.1.1. Внедрить SSH в production
**Зачем:** Telnet明文 — нарушение базовой безопасности. Huawei MA5608T поддерживает SSHv2.  
**Как:**
1. Расширить `core/olt.py` `OltConnection` SSH-транспортом (paramiko или asyncssh).
2. Добавить `ssh: bool` в `config.yaml` per-OLT.
3. Feature flag для миграции: `use_ssh` alongside `use_telnet`.

### 4.1.2. CI/CD пайплайн (GitHub Actions)
**Зачем:** smoke-тесты не запускаются на каждый PR. Рефакторинг `parser.py` = поломка парсинга в production.  
**Как:**
```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
jobs:
  lint-test:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - run: uv sync --dev
      - run: uv run python -m tests.test_smoke
      - run: uv run ruff check core/ tests/
      - run: uv run mypy core/
```

### 4.1.3. Secrets rotation через env + python-dotenv
**Зачем:** `.env` на диске — риск утечки.  
**Как:**
- Перевести на `dotenv` с перезагрузкой (watchdog) или integrate с OS keychain (Windows Credential Manager / Linux Secret Service).
- Для Docker/MCP — использовать `--env-file` или враппер, который читает из Vault в рантайме.

### 4.1.4. Graceful degradation при сбоях Telnet
**Зачем:** Если OLT перегружен, инструмент падает.  
**Как:**
- Exponential backoff в `core/olt.py` send_command (сейчас `time.sleep` фиксированный).
- Circuit breaker per OLT: 3 ошибки подряд → минус OLT из автопоиска на N минут.

## Medium Priority (Reliability & Observability)

### 4.2.1. Structured logging + Correlation ID
**Зачем:** Сейчас `logging.basicConfig` с format `%(levelname)s: %(message)s`. Невозможно trace запрос через parser → engine → report.  
**Как:**
- `structlog` или `loguru` с JSON-форматом.
- Генерация `trace_id` в `diagnose.py:run_diagnosis` и проброс через thread-local context.

### 4.2.2. Prometheus metrics для внутренних операций
**Зачем:** Нет цифр. Сколько диагностик в день? Средний latency? Сколько OLT недоступно?  
**Как:**
- `prometheus-client` → `/metrics` endpoint.
- Метрики: `diagnosis_total{olt,status}`, `olt_duration_seconds`, `parser_errors_total{pattern}`, `rule_fires_total{rule}`.

### 4.2.3. Port snapshot diffing (temporal analytics)
**Зачем:** Сейчас сохраняются `PortSnapshot`, но не сравниваются.  
**Как:**
- Построить `diff` между `t-1` и `t`: новые offline ONT, ухудшение Rx, появление ошибок.
- Автоматически генерировать `DiagnosisProblem` для всего порта (bulk mode).

### 4.2.4. Модулярность pytest + parametrize
**Зачем:** Smoke-тесты — монолит `test_smoke.py`. Нет параметризации под разные модели ONT.  
**Как:**
- `tests/unit/parser/test_optical.py`
- `tests/unit/engine/test_rules.py` с `@pytest.mark.parametrize` для каждой границы thresholds.
- `tests/integration/test_olt_adapter.py` с `responses` или моками на socket.

## Low Priority (DX & Future-Proofing)

### 4.3.1. Helm Chart / Kustomize для Web-компонентов
**Зачем:** Web UI + DB + scripts можно упаковать.  
**Как:**
- `web/Dockerfile` (отдельный, multi-stage).
- `deployment/k8s/` с Deployment + Service + Ingress.
- Убрать `check_and_start_server.py` в initContainer.

### 4.3.2. Parser-as-Code
**Зачем:** `PATTERNS` dict — хрупкий. При изменении CLI Huawei нужно обновлять regex вручную.  
**Как:**
```python
@dataclass
class FieldSpec:
    name: str
    pattern: str
    transformer: Callable[[str], Any] = str

FIELDS = [
    FieldSpec("ont_rx_power", r"Rx optical power\(dBm\)\s*:\s*(-?[\d.]+)", float),
    ...
]
```
Или JSON Schema для вывода Huawei.

### 4.3.3. Офлайн-режим для MCP
**Зачем:** AI-агент не должен требовать live-доступ к OLT.  
**Как:**
- `gpon_diagnose` с параметром `mock_data`. Если `mock=True` — загрузить `data/samples/` last-known-good.
- HTML-отчёты с timeline графиками (Plotly) можно генерировать без OLT.

### 4.3.4. Distroless / minimal Docker image
**Зачем:** Сейчас `python:3.12-slim` + `curl` = ~170MB. Для MCP-сервера это не критично, но можно уменьшить.  
**Как:**
- `python:3.12-alpine` или distroless + uv-слой.
- Multi-stage: builder с `uv` → runner только `.venv`.

### 4.3.5. Конфигурация через Pydantic Settings
**Зачем:** `config.yaml` + `.env` — два источника. Нет валидации.  
**Как:**
```python
class OltConfig(BaseSettings):
    name: str
    host: str
    port: int = 23
    credential_key: str = "RADIUS"

class Settings(BaseSettings):
    olts: list[OltConfig]
    thresholds: Thresholds
    bad_versions: list[str]
```
Автоматическая валидация типов + дефолты.

### 4.3.6. Type-safe output (Pydantic / dataclass для парсера)
**Зачем:** `parse_*` заполняют `OntMetrics` вручную. Нет гарантий, что все поля проинициализированы.  
**Как:** 
- Pydantic `model_validator(mode="after")` для проверки sentinel-логики.
- Строгий `mypy --strict` с `from __future__ import annotations`.
