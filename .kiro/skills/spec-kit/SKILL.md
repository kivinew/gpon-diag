# Spec Kit — 6 фраз для любых ИИ-агентов

Шесть универсальных практик, которые снижают расход токенов и повышают качество результата.
Применимы к любому агенту: Pi, Kiro, Claude, Opencode, Mimocode, Cursor, Copilot, Codex, Qwen и пр.

---

## Фраза 1: Запускай субагентов / параллельные задачи
**Когда использовать:** задача содержит 3+ независимых шага

**Как реализовать:**
- Kiro / Hermes: `invoke_sub_agent` / `delegate_task`
- Claude Projects: несколько отдельных чатов
- Cursor / Copilot: отдельные composer-сессии
- Любой агент: явно разбить задачу на независимые части и дать каждую отдельным запросом

```
Субзадача А: [описание, контекст, входные данные]
Субзадача Б: [описание, контекст, входные данные]
Субзадача В: [описание, контекст, входные данные]
```

**Правило:** если запрос >300 слов или затрагивает 3+ независимых компонента — предложить разбить на субзадачи.

**Ограничение:** не запускать параллельно до утверждения спецификации. Иначе агенты решают задачу независимо на неполных данных — расхождение гарантировано.

## Фраза 2: Напиши implementation spec (спецификацию)
**Когда использовать:** перед любой нетривиальной задачей

**Что включить в спецификацию:**
1. Цель и критерий готовности
2. Шаги реализации с порядком
3. Принятые технические решения и обоснование
4. Граничные случаи и ограничения

**Как запросить у агента:**
```
Перед тем как писать код, составь спецификацию:
- Цель задачи
- Порядок шагов
- Технические решения
- Критерий готовности
Жди моего подтверждения перед реализацией.
```

**Агент считает спецификацию утверждённой** только при явном «да» / «согласовано» / «приступай». «Выглядит нормально» или молчание — не подтверждение.

## Фраза 3: Проинтервьюируй меня
**Когда использовать:** перед спецификацией, если требования нечёткие

**Вопросы, которые агент должен задать:**
1. Цель: кто пользователи, какую проблему решаем
2. Ключевые решения: архитектура, библиотеки, хранилище данных
3. Человек в контуре: где нужен явный контроль (деплой, удаление, финансы)
4. Критерий завершённости: как поймём, что готово

**Масштаб интервью:**
- Тривиальная задача (1 атомарное действие): 0 вопросов, выполнять немедленно
- Средняя задача (1 функциональность): ≤3 вопроса
- Крупная задача (несколько компонентов): полное интервью

**Как запросить:**
```
Сначала проинтервьюируй меня — задай до 3 вопросов,
которые помогут составить точную спецификацию.
```

## Фраза 4: Проверяй до того, как строить (smoke test)
**Когда использовать:** после любых изменений в коде

**Универсальные шаги верификации:**
```bash
# Python-проекты
uv run python -m tests.test_smoke
# или
python -m pytest tests/ -x -q

# Node-проекты
npm test -- --run

# Любой проект: запустить линтер
ruff check .      # Python
eslint src/       # JS/TS
```

**Агент должен:**
1. Объявить метод проверки ДО запуска: «Буду проверять через X»
2. Выполнить проверку и зафиксировать результат
3. Исправить ошибки до передачи результата пользователю

**Ручная проверка (зона ответственности пользователя):**
- Деплой в production
- Удаление данных / миграции
- Финансовые операции
- Отправка уведомлений

## Фраза 5: Сделай из этого скилл / стиринг
**Когда использовать:** удачный паттерн повторился 2+ раза

**Что упаковать:**
- Успешная последовательность шагов
- Принятые решения и их обоснование
- Known issues и способы их обхода

**Форматы для разных агентов:**

| Агент | Формат | Путь |
|-------|--------|------|
| Kiro | `.kiro/steering/*.md` или `.kiro/skills/*/SKILL.md` | workspace или `~/.kiro/` |
| Hermes | `.hermes/skills/*/SKILL.md` | workspace |
| Claude | Custom instructions / Project knowledge | настройки проекта |
| Cursor | `.cursor/rules/*.mdc` | workspace |
| Любой | `AGENTS.md` / `CLAUDE.md` / `COPILOT.md` | корень репозитория |

**Структура скилла:**
```markdown
# Название навыка
Когда применять (триггеры).
## Шаги
1. ...
## Known Issues
- Проблема → решение
```

## Фраза 6: Автоматизируй это
**Когда применять:** только детерминированные, измеримые операции

**Критерии автоматизируемости:**
- ✅ Результат однозначен и проверяем
- ✅ Нет оценочных суждений в процессе
- ✅ Нет необратимых последствий (или они допустимы)
- ❌ Результат зависит от «вкуса» — только аугментация (агент предлагает, человек решает)

**Инструменты по агентам:**

| Агент | Событийная автоматизация | Периодическая |
|-------|--------------------------|---------------|
| Kiro | Hooks (fileEdited, agentStop, preToolUse…) | — |
| Hermes | `cronjob(schedule=…)` | `cronjob(schedule="0 9 * * *")` |
| GitHub Actions | `on: push / pull_request` | `on: schedule: cron:` |
| Любой | webhook + CI/CD pipeline | cron + запуск агента по расписанию |

**Пример Kiro-хука:**
```json
{
  "name": "Run tests on save",
  "version": "1.0.0",
  "when": { "type": "fileEdited", "patterns": ["**/*.py"] },
  "then": { "type": "runCommand", "command": "uv run python -m tests.test_smoke" }
}
```

## Gotchas — ошибки, которых стоит избегать

| Ошибка | Последствие | Решение |
|--------|------------|---------|
| Параллельные субагенты без утверждённой спецификации | Расходящиеся результаты, больше правок чем выигрыш | Сначала спека, потом параллелизм |
| Интервью пропущено | Реализация не того, что нужно | Всегда ≥1 уточняющий вопрос для нетривиальных задач |
| Smoke test пропущен | Баги обнаружены на приёмке | Верификация до передачи результата |
| Автоматизация процессов с оценочными суждениями | Непредсказуемый результат | Режим аугментации: агент предлагает, человек принимает |
| Скилл без раздела known issues | Повторение одних и тех же ошибок | Known issues обязательны при упаковке паттерна |
| Молчание принято за подтверждение спецификации | Реализация неутверждённого плана | Явное «да» или «приступай» — единственное подтверждение |
