# spec-pilot — методология структурированной разработки с AI-агентами

Методология определяет обязательный порядок работы: **сбор требований → спецификация → реализация → верификация**. Соблюдение порядка исключает избыточный расход контекста на итерации по неопределённым требованиям.

---

## Правильный порядок (обязателен)

| Шаг | Действие | Назначение |
|-----|----------|------------|
| 1 | Интервью | Извлечение требований через целевые вопросы — без этого шага исходные данные для спецификации неполны |
| 2 | Спецификация | Фиксация принятых решений в явном виде и получение подтверждения пользователя — исключает неоднозначность при реализации |
| 3 | Делегирование | Распределение задач по спецификации — допустимо только после явного утверждения спецификации |
| 4 | Верификация | Проверка результата до передачи — обнаруживает несоответствия до приёмки |

Порядок шагов изменять нельзя.

Шаги 5–6 находятся за пределами конвейера реализации: **упаковка в навык** (повторяемые паттерны + раздел known issues) и **автоматизация** (применяется только к измеримым, детерминированным операциям — см. раздел «Типичные ошибки»).

---

## Ограничения (применяются всегда)

### Ограничение 1 — Оценка масштаба перед запуском

Перед запуском конвейера необходимо определить масштаб задачи:

- **Тривиальная задача** (одно атомарное действие с очевидным решением): конвейер не запускается, задача выполняется немедленно без уточняющих вопросов.
- **Задача среднего масштаба** (одна функциональность, 2–4 неочевидных технических решения): интервью не более 3 вопросов → краткая спецификация → реализация → верификация.
- **Крупная задача** (несколько подсистем или взаимозависимых компонентов): полное интервью → детальная спецификация → параллельное делегирование → верификация.

При неопределённости масштаба применяется более высокий уровень. Снижение уровня допустимо только по явному указанию пользователя.

### Ограничение 2 — Запрет параллельного делегирования без утверждённой спецификации (критическое)

Параллельное выполнение без общей спецификации приводит к тому, что каждый агент решает задачу независимо на основе неполных данных. Результаты расходятся, объём требуемых исправлений превышает исходный выигрыш от параллелизации.

- Спецификация считается утверждённой только при **явном подтверждении** пользователя в ответ на прямой вопрос: «Спецификация согласована? Приступаю к реализации?»
- Ответы «выглядит нормально», «делай как считаешь нужным», а также молчание подтверждением не являются — требуется переспросить.
- Запрет распространяется на любое параллельное делегирование, включая разведочные подзадачи на этапе интервью.
- Если пользователь запрашивает параллельное выполнение при отсутствии спецификации — запрос отклоняется с предложением сначала сформировать спецификацию.
- При отклонении спецификации — возврат на шаг 2, реализация не начинается.

### Ограничение 3 — Зоны ответственности пользователя (необратимые операции)

Следующие операции агент не выполняет самостоятельно — только с явным подтверждением пользователя:

- финансовые операции и передача платёжных данных;
- удаление данных, миграции баз данных;
- отправка уведомлений и рассылок;
- деплой в производственную среду;
- операции с необратимыми последствиями: `git push --force`, `rm -rf`, массовые изменения данных;
- использование боевых ключей API и секретов;
- webhooks и интеграции с внешними системами в production.

При любом сомнении операция классифицируется как необратимая и требует подтверждения.

### Ограничение 4 — Управление расходом контекста

- Параллельные агенты расходуют общий контекстный лимит сессии — не меньше, чем последовательное выполнение.
- Методология снижает расход за счёт устранения итераций по неопределённым требованиям, но не увеличивает доступный лимит.
- Задачи следует декомпозировать только на действительно независимые части. Избыточное дробление увеличивает накладные расходы.

## Алгоритм выполнения

**Шаг 1 — Интервью.** В соответствии с оценённым масштабом (Ограничение 1) задаются целевые вопросы, раскрывающие ключевые технические решения: структура данных, способ хранения, граничные случаи, критерий завершённости. Если пользователь делегирует выбор агенту — принятые допущения озвучиваются явно.

**Шаг 2 — Спецификация.** Формируется документ: цель, шаги реализации, принятые технические решения, критерий готовности. Спецификация предъявляется пользователю и ожидается явное подтверждение (Ограничение 2). При несогласии — уточнение, реализация не начинается.

**Шаг 3 — Реализация (только после подтверждения спецификации).** Спецификация декомпозируется на независимые части. По умолчанию — последовательное выполнение. Параллельное делегирование применяется только при наличии действительно независимых блоков и доступности механизма параллелизации.

**Шаг 4 — Верификация.** Три уровня:
1. Декларация метода проверки до начала верификации: «Опишу, как буду проверять, затем выполню проверку — тесты, линтер, браузер».
2. Автоматическая проверка: тесты, линтер, ручной просмотр в браузере — агент фиксирует собственный результат.
3. Зоны ответственности пользователя (Ограничение 3): финансовые операции, удаление данных, производственная среда — проверяются пользователем вручную.

**Шаг 5 (опционально) — Упаковка в навык.** При наличии повторяемого и воспроизводимого паттерна предлагается оформить его в навык с обязательным разделом known issues.

## Типичные ошибки

| Симптом | Корректное действие |
|---------|---------------------|
| Параллельное выполнение запущено без подтверждения спецификации | Остановить выполнение, вернуться на шаг 2. Команда пользователю: «стоп, сначала спецификация» |
| Интервью содержит избыточное число вопросов | Ограничение 1: для задачи среднего масштаба — не более 3 вопросов, для тривиальной — 0 |
| Параллельное делегирование недоступно или нестабильно | Перейти к последовательному выполнению по той же спецификации — семантика и порядок сохраняются |
| Запрос на автоматизацию произвольного процесса | Автоматизация применима только к детерминированным измеримым операциям. Операции, требующие оценочных суждений, выполняются в режиме дополнения (агент предлагает, пользователь принимает решение). Агент не исключается из контура управления на необратимых или дорогостоящих операциях |

---

## Гарантии методологии

- Реализация не начинается до получения полных требований и явного подтверждения спецификации.
- Параллельное делегирование не запускается без утверждённой спецификации.
- Необратимые операции не выполняются автономно.
- Конвейер не запускается для тривиальных задач.
- Реальный выигрыш от применения методологии: **3–5× снижение итераций** на задачах среднего и высокого масштаба за счёт устранения неопределённости на старте.
